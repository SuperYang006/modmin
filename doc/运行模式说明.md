# Modmin 运行模式说明

本文档用于说明当前项目支持的三种运行模式：

1. `mock`
2. `http`
3. `tcb`

并明确各模式的定位、适用场景与限制，避免后续误将三者视为同等能力。

## 1. 结论

当前项目虽然同时支持 `mock / http / tcb` 三种模式，但它们并不是同等优先级：

1. `http` 是当前推荐的本地开发模式。
2. `tcb` 是当前推荐的真实云端联调模式。
3. `mock` 只适合轻量 UI 演示或极早期静态联调，不是主开发模式。

## 2. 模式说明

### 2.1 `mock`

含义：

1. 前端不请求真实云函数。
2. 使用本地 mock 数据与 mock handler 返回结果。

适用场景：

1. 页面静态演示。
2. 极早期 UI 草图验证。
3. 不依赖真实权限、真实登录、真实数据库结果的简单交互调试。

限制：

1. 无法真实还原 CloudBase 登录链路。
2. 无法完整验证 HTTP 触发器行为。
3. 无法可靠验证审计日志、Webhook、定时触发器等依赖真实后端能力的功能。
4. 随着项目能力增多，维护一套长期正确的 mock 分支成本较高。

结论：

1. `mock` 不是当前项目的主开发模式。
2. 不建议把复杂功能是否可用建立在 `mock` 模式之上。

### 2.2 `http`

含义：

1. 前端通过 HTTP 请求访问本地 `local-server`。
2. `local-server` 在本机直接加载并执行云函数源码。
3. 数据库通常仍连接真实 CloudBase 环境。

适用场景：

1. 本地开发。
2. 功能联调。
3. 云函数行为调试。
4. 审计日志、Webhook、权限、CRUD 等真实后端链路验证。

优点：

1. 更贴近真实业务执行逻辑。
2. 可以复用本地代码直接调试云函数。
3. 可以补充 HTTP 请求头、IP、UA 等本地运行信息。
4. 比 `mock` 更容易发现真实后端问题。

限制：

1. 仍然不是完整线上环境。
2. 某些只存在于真实 CloudBase 运行时的行为需要 `tcb` 模式进一步验证。

结论：

1. `http` 是当前推荐的本地开发模式。

### 2.3 `tcb`

含义：

1. 前端直接使用 CloudBase JS SDK。
2. 登录成功后通过 `callFunction` 调云端函数。
3. 首次登录仍需通过 `modmin_auth` 的 HTTP 触发器获取 Ticket。

适用场景：

1. 真实云端联调。
2. 发布前验证。
3. CloudBase 运行时行为确认。

优点：

1. 最接近真实线上环境。
2. 可以验证云函数部署产物和云端依赖是否完整。
3. 能暴露本地 `http` 模式无法完全模拟的问题。

限制：

1. `callFunction` 事件默认拿不到真实客户端 HTTP 请求头。
2. 实时客户端 IP 在非 HTTP 触发器链路中通常不可得。
3. 联调成本高于 `http` 模式。

结论：

1. `tcb` 是当前推荐的真实云端联调模式。

## 3. 为什么 `mock` 不友好

当前项目的整体能力与 CloudBase 运行时贴合较深，因此 `mock` 天然不适合作为主模式。

主要原因：

1. 登录链路依赖 CloudBase 自定义登录与 Ticket 流程。
2. 业务能力依赖云函数调用语义。
3. 审计日志、Webhook、定时处理、函数间调用等能力不容易被完整 mock。
4. 为复杂系统长期维护一套“半真实”的 mock 分支成本过高。

因此：

1. `mock` 更适合作为辅助模式。
2. `http` 和 `tcb` 才是当前项目的主运行模式。

## 4. 推荐使用方式

### 日常开发

推荐使用：

```text
http
```

### 发布前联调

推荐使用：

```text
tcb
```

### 纯界面演示

可选使用：

```text
mock
```

## 5. 文档建议口径

后续在 README、部署文档、开源说明中，建议统一使用以下口径：

1. `mock`：仅用于有限 UI 演示，不保证完整功能。
2. `http`：推荐本地开发模式。
3. `tcb`：推荐真实云端联调模式。

## 6. 当前判断

当前项目不应继续追求让 `mock` 与 `http / tcb` 完全等价。

更合理的方向是：

1. 明确 `mock` 的弱化定位。
2. 持续强化 `http` 与 `tcb` 两条真实链路。
