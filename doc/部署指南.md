## 1. 前置要求

- 微信云开发（CloudBase）环境一个，已开通数据库、云函数、静态托管（开通环境后静态托管需要确认一下是否需要手动确认开启）
- Node.js 18+
- 项目根目录 `npm install`，会自动装上 `@cloudbase/cli`、`vitest`、`jsonwebtoken`
- `openssl`（用于生成 JWT 密钥；macOS / 主流 Linux 默认带）

> 仓库根 `package.json` 已声明 `@cloudbase/cli`，无需额外全局安装。所有部署命令通过 `npm run` 调用。

## 2. 第一步：准备 CloudBase 环境

### 2.1 获取自定义登录私钥

Modmin 使用 CloudBase 自定义登录方式，需要一份私钥文件来签发登录凭证。

1. 打开 [CloudBase 控制台](https://console.cloud.tencent.com/tcb)
2. 进入你的环境 → **用户管理** → **登录方式**
3. 开启「自定义登录」
4. 下载私钥文件，重命名为 `tcb_custom_login.json`
5. 将文件放置到 `cloudfunctions/modmin_auth/` 目录下

> 建议在这里补一张截图：展示 CloudBase 控制台中“用户管理 -> 登录方式 -> 自定义登录”的入口位置，避免初次接入时找不到页面。
>
> 建议图片占位：
> `doc/images/deploy/cloudbase-custom-login.png`

文件结构示例：
```json
{
  "private_key_id": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
  "private_key": "-----BEGIN RSA PRIVATE KEY-----\n...\n-----END RSA PRIVATE KEY-----\n",
  "env_id": "your-env-id"
}
```

> **安全提示**：`tcb_custom_login.json` 已在 `.gitignore` 中忽略，**不要提交到代码仓库**。

### 2.2 生成 JWT 签名密钥

5 个云函数用同一个 `MODMIN_JWT_SECRET` 签发 / 验证登录 token。生产环境必须设置，且与本地开发密钥不同。

```bash
openssl rand -hex 32
```

把输出的 64 字符串保存到你的密码管理器。后续：

- **云端部署**：`export MODMIN_JWT_SECRET=<串>` 后跑 `npm run deploy:fn`，CLI 会把它注入 5 个函数的环境变量（详见 4.2）
- **本地开发**：写入 `local-server/cloudbase.local.json` 的 `jwtSecret` 字段（详见附录 C）

> ⚠️ 密钥一旦轮换，所有已签发的 access / refresh token 立即失效，已登录用户会被踢回登录页。建议选低峰期操作，且**生产和本地用不同的密钥**。

### 2.3 CloudBase 环境信息定位

建议在部署文档中补一张截图，说明如何在 CloudBase 控制台定位：

1. 当前环境名称
2. `envId`
3. 静态托管与数据库入口

这样可以帮助陌生用户更快确认自己正在操作正确的环境。

> 建议图片占位：
> `doc/images/deploy/cloudbase-env-overview.png`

## 3. 第二步：创建系统集合

如果你使用 `npm run setup`，以下集合会由脚本自动检查并创建；如果你走手工部署流程，则需要在 CloudBase 控制台的「数据库」中手动创建以下集合（集合名区分大小写）：

| 集合名 | 用途 | 是否必须 |
| --- | --- | --- |
| `modmin_collections` | 模型定义与字段配置（系统核心） | ✅ 必须 |
| `modmin_admin_users` | 管理后台账号 | ✅ 必须 |
| `modmin_admin_roles` | 管理后台角色（含内置角色，部署初始化阶段自动补齐） | ✅ 必须 |
| `modmin_role_permissions` | 角色与业务模型的增删改查权限矩阵 | ✅ 必须 |
| `modmin_sessions` | 登录 refresh token 会话 | ✅ 必须 |
| `modmin_menu_groups` | 菜单分组（二级分组，可选启用） | ✅ 必须 |
| `modmin_audit_logs` | 操作审计日志 | ✅ 必须 |
| `modmin_webhooks` | Webhook 配置 | ✅ 必须 |
| `modmin_webhook_deliveries` | Webhook 投递记录 | ✅ 必须 |

> 集合创建后无需添加任何初始数据。`modmin_admin_roles` 的内置角色会在部署初始化阶段自动补齐。

**手工创建步骤**（以控制台操作为例）：

1. 进入 CloudBase 控制台 → **数据库**
2. 点击「新建集合」
3. 依次输入上表中的集合名，点击确定
4. 重复以上步骤创建全部 9 个集合

> 建议在这里补一张截图：展示“数据库 -> 新建集合”的按钮位置和控制台界面。
>
> 建议图片占位：
> `doc/images/deploy/cloudbase-create-collection.png`

补充说明：

1. 当前项目初始化脚本使用 `@cloudbase/node-sdk` 的 `db.createCollection()` 创建集合。
2. 如果你使用 `npm run setup`，出现“集合未创建成功”时，优先检查：
   - CloudBase 凭据是否有当前环境数据库管理权限
   - 环境 ID 是否正确
   - 集合创建后是否存在短暂可见性延迟

> 集合前缀由云函数环境变量 `MODMIN_COLLECTION_PREFIX`（默认 `modmin_`）控制。除非要在同一环境与其它业务隔离，否则保持默认即可，文档中所有集合名都以此为前提。

## 4. 第三步：部署云函数

项目当前共有 7 个云函数，全部位于 `cloudfunctions/` 目录下：

| 目录名 | 对应云函数名 |
|---|---|
| `modmin_auth/` | `modmin_auth` |
| `modmin_audit/` | `modmin_audit` |
| `modmin_crud/` | `modmin_crud` |
| `modmin_runtime/` | `modmin_runtime` |
| `modmin_schema/` | `modmin_schema` |
| `modmin_system/` | `modmin_system` |
| `modmin_webhook/` | `modmin_webhook` |

### 4.1 使用微信开发者工具上传

1. 打开微信开发者工具，关联你的 CloudBase 环境
2. 在云函数面板中，逐个右键点击上述 7 个目录
3. 选择「上传并部署：云端安装依赖」
4. 部署后，进入控制台为每个函数配置 [环境变量](#43-环境变量)

### 4.2 使用 CloudBase CLI 一键部署（推荐）

CLI 方式可一次推送 7 个函数 + 环境变量，避免控制台逐个点击。

**首次准备：**

```bash
# 项目根目录已经在 devDependencies 中声明 @cloudbase/cli
npm install

# 生成临时部署密钥（仅首次，本地必须保存好）
openssl rand -hex 32   # 输出形如：1d8a9b8...，作为 MODMIN_JWT_SECRET

# 登录 CloudBase（浏览器扫码授权）
npm run tcb:login
```

**每次部署：**

```bash
# 在 shell 中提供 JWT 密钥（不要写进 .env，避免误提交）
export MODMIN_JWT_SECRET=<上一步生成的随机串>

# 一键部署全部 5 个函数（自动生成 cloudbaserc.json 并推送 env）
npm run deploy:fn

# 或只部署单个函数
npm run deploy:fn:single -- modmin_auth
```

部署流程：

1. `scripts/build-cloudbaserc.js` 从环境变量渲染出 `cloudbaserc.json`（已 gitignore）
2. `tcb fn deploy --all --force --yes` 按配置批量上传 + 设置环境变量

> ⚠️ `MODMIN_JWT_SECRET` 不进入仓库。`cloudbaserc.json` 含明文密钥，已加入 `.gitignore`。

#### 根目录 npm scripts 速查

仓库根 `package.json` 提供以下命令，全部在仓库根目录执行：

| 命令 | 实际执行 | 说明 |
| --- | --- | --- |
| `npm test` | `vitest run` | 跑一次 [tests/](../tests/) 下全部用例后退出，CI 用这个 |
| `npm run test:watch` | `vitest` | 监听模式：保存文件就增量重跑相关用例 |
| `npm run tcb:login` | `tcb login` | 调起浏览器扫码登录 CloudBase，凭据存到 `~/.config/.cloudbase/auth.json` |
| `npm run deploy:prepare` | `node scripts/build-cloudbaserc.js` | 从 `MODMIN_ENV_ID`、`MODMIN_JWT_SECRET` 渲染出 `cloudbaserc.json`；缺任一必填变量时直接报错退出。一般不直接调，作为部署前置步骤 |
| `npm run deploy:fn` | `deploy:prepare && tcb fn deploy --all --force --yes` | 全量部署 5 个云函数 + 注入环境变量 |
| `npm run deploy:fn:single -- <name>` | `deploy:prepare && tcb fn deploy --force --yes <name>` | 只部署指定函数，例：`npm run deploy:fn:single -- modmin_auth` |
| `npm run build:web` | `npm --prefix web-admin run build` | 构建前端，产物输出到 `web-admin/dist/` |
| `npm run deploy:web` | `node scripts/deploy-web.js` | 交互式部署前端：列出现有顶级目录供选择，或新建目录、或部署到根目录；选定后自动构建 + 上传 |
| `npm run deploy:all` | `deploy:fn && deploy:web` | 一次性发全套：先云函数后前端 |

`&&` 串联意味着前一步失败（比如缺 `MODMIN_JWT_SECRET`）就不会执行下一步，避免推上空环境变量。`--` 之后的参数原样追加给底层 `tcb` 命令。

> 子项目（`web-admin/`、`local-server/`、`cloudfunctions/`）有各自的 `package.json` 与 npm scripts；根 scripts 只负责 **测试** 与 **部署** 这两件横跨整仓的事。

### 4.3 环境变量

5 个云函数都依赖以下环境变量（CLI 方式自动注入；控制台方式需手工为每个函数配置）：

| 变量 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `MODMIN_JWT_SECRET` | ✅ | — | JWT 签名密钥，**至少 32 字符**，缺失或长度不足时云函数拒绝启动 |
| `MODMIN_COLLECTION_PREFIX` | ❌ | `modmin_` | 系统集合名前缀，仅在与其他业务共用环境时需要改 |
| `MODMIN_ENV_ID` | ✅ | — | CloudBase 环境 ID，CLI 部署时必填 |

**密钥轮换：** 修改后所有已签发的 access/refresh token 立即失效，已登录用户被踢回登录页，建议选低峰期操作。

### 4.4 配置 `modmin_auth` HTTP 触发器

`modmin_auth` 云函数需要额外开启 HTTP 访问，用于前端首次登录（CloudBase 登录态建立前无法使用 `callFunction`）。

1. 进入 CloudBase 控制台 → **云函数** → `modmin_auth`
2. 点击「HTTP访问服务」→ 开启
3. 记录生成的访问地址，例如：
   ```
   https://your-env-id-xxxxxxxxxx.ap-shanghai.app.tcloudbase.com/modmin_auth
   ```
   后续前端配置需要用到这个地址。

>
> 1. `modmin_auth` 云函数页面中“HTTP访问服务”的开启入口
> 2. 开启后生成的实际访问地址位置
>
> 建议图片占位：

## 5. 第四步：配置并部署前端

### 5.1 配置环境变量

复制 `web-admin/.env.production.example` 为 `web-admin/.env.production.local`，填入你的环境信息。该文件只用于本机生产构建，不提交到仓库：

```bash
cp web-admin/.env.production.example web-admin/.env.production.local
```

编辑 `web-admin/.env.production.local`：

```env
VITE_API_MODE=tcb
VITE_MODMIN_ENV_ID=your-env-id              # CloudBase 环境 ID
VITE_MODMIN_REGION=ap-shanghai              # 环境所在地域
VITE_MODMIN_FUNCTION_PREFIX=modmin_         # 云函数名前缀，保持默认即可
VITE_MODMIN_AUTH_LOGIN_URL=https://...      # 第 4.4 步获取的 HTTP 访问地址
VITE_BASE_PATH=/                            # 部署子路径，根目录填 /，子目录如 /modmin/
```

> ⚠️ `VITE_BASE_PATH` 必须与第 5.3 步的部署目录严格一致：根目录部署填 `/`，部署到 `/modmin/` 子目录就填 `/modmin/`。`deploy:web` 脚本会在上传前自动校验，不一致直接拒绝部署并提示该改成什么值。

### 5.2 构建前端

```bash
cd web-admin
npm install
npm run build
```

构建产物在 `web-admin/dist/` 目录。

### 5.3 部署到静态托管

#### 方式一：CloudBase CLI（推荐）

`npm run deploy:web` 会启动一个交互流程：

1. 调 `tcb hosting list` 列出当前环境下静态托管的所有顶级目录
2. 让你选择部署目标：
   - `0)` 部署到根目录 `/`
   - `1)` ~ `n)` 已有的子目录（直接覆盖）
   - `n+1)` 新建目录…（输入新目录名）
3. 二次确认后跑 `npm run build:web` 构建，再 `tcb hosting deploy` 上传

```bash
# 确保已登录 CloudBase（与部署云函数共用同一套凭据）
npm run tcb:login

# 交互式部署
npm run deploy:web

# 或者把云函数 + 前端一起发（前端在最后一步）
npm run deploy:all
```

**非交互模式**（CI / 脚本调用）：

```bash
# 跳过交互直接部署到指定目录
npm run deploy:web -- --path admin

# 部署到根目录
npm run deploy:web -- --path /

# dist 已就绪时跳过构建
npm run deploy:web -- --path admin --skip-build

# 上传前先清空目标目录（避免 Vite hash 资源越积越多）
# 注意：仅在子目录部署时生效，根目录会自动忽略以保护同环境其它子站点
npm run deploy:web -- --path admin --clean

# 只构建 + 校验，不真正上传（排查 base path 等问题）
npm run deploy:web -- --path admin --dry-run
```

环境 ID 由 `MODMIN_ENV_ID` 显式提供。部署到子目录时记得把 `VITE_BASE_PATH` 改成对应路径再构建（参见 §5.1）。

**前置校验**：脚本会在上传前对比 `dist/index.html` 里的资源前缀与目标目录，如果不一致（如选 `modmin/` 但构建产物指向 `/admin/`）直接拒绝部署并提示该改 `VITE_BASE_PATH` 为哪个值。这能避免"部署完打开是白屏 / 资源 404"的常见坑。

#### 方式二：控制台手工上传

1. 进入 CloudBase 控制台 → **静态托管**
2. 将 `web-admin/dist/` 下的所有文件上传到托管根目录（或你指定的子目录）
3. 如果部署到子目录（如 `/modmin/`），需将 `VITE_BASE_PATH` 设置为对应路径后重新构建

> SPA 路由配置：静态托管需开启「错误页面重定向」，将 404 指向 `index.html`，否则刷新页面会报错。若使用前端路由并部署在子路径下，也要确保构建时 `VITE_BASE_PATH` 与实际托管路径一致。

## 6. 第五步：创建初始超管 + 验证安装

### 6.1 创建超级管理员账号

`modmin_admin_users` 集合默认为空，登录页无法直接进入。需要先手工写入一条 `role_super_admin` 记录。

#### 方式一：CloudBase 控制台手工添加

进入「数据库 → `modmin_admin_users` → 新增记录」，填入：

| 字段 | 值 | 说明 |
| --- | --- | --- |
| `userName` | `admin` | 登录用账号 |
| `nickName` | `超级管理员` | 显示名 |
| `roleCode` | `role_super_admin` | 必须是这个值 |
| `status` | `enabled` | 启用 |
| `passwordSalt` | 任意 16 字符随机串 | 用 `openssl rand -hex 8` 生成 |
| `passwordHash` | `sha256(密码 + passwordSalt)` 的十六进制 | 见下方一行命令 |
| `createTime` | `Date.now()` 的输出（毫秒时间戳） | 可省略 |

生成 `passwordHash` 的一行命令（替换 `YourPassword` 与上面的 salt）：

```bash
node -e 'console.log(require("crypto").createHash("sha256").update("YourPassword" + "your_salt_here").digest("hex"))'
```

#### 方式二：脚本一次性写入（推荐，避免手抖）

```bash
node -e '
const cb = require("@cloudbase/node-sdk")
const crypto = require("crypto")
const { secretId, secretKey, envId } = require("./local-server/cloudbase.local.json")
const app = cb.init({ secretId, secretKey, env: envId })
const db = app.database()
const salt = crypto.randomBytes(8).toString("hex")
const password = "ChangeMe@2026"
const passwordHash = crypto.createHash("sha256").update(password + salt).digest("hex")
db.collection("modmin_admin_users").add({
  userName: "admin",
  nickName: "超级管理员",
  roleCode: "role_super_admin",
  status: "enabled",
  passwordSalt: salt,
  passwordHash,
  createTime: Date.now(),
}).then((r) => console.log("created:", r.id, "password:", password))
'
```

> ⚠️ 这条命令直接打到生产环境数据库，请确认 `cloudbase.local.json` 指向的就是生产 envId，并把日志里的密码立即换成强密码、保存到密码管理器。

### 6.2 验证

1. 打开第 5.3 步部署好的前端地址
2. 用刚创建的 `admin` 账号登录
3. 进入「角色管理」页面，确认内置角色已存在
4. 进入「模型管理」点击「新建模型」，能正常保存即表示部署成功

### 6.3 后续运维

- **新增管理员**：在「用户管理」页面建账号，默认绑定 `role_operator`，需要超管去「角色管理 → 配置权限」给该角色分配业务模型权限
- **业务模型权限**：所有非超管账号默认看不到任何业务模型；超管在「角色管理 → 配置权限」按 `(roleCode, collectionName)` 粒度勾选 canList / canCreate / canUpdate / canDelete
- **角色停用**：把 `modmin_admin_roles` 里某个 roleCode 的 `status` 改成 `disabled` 后，绑定该角色的所有用户立即无法登录、无法读写任何业务数据（已修复，参见附录 B `modmin_auth` / `modmin_crud` 的鉴权说明）

## 附录 A：集合说明

### `modmin_collections`

**这是系统最核心的集合**，存储所有通过「模型管理」配置的数据模型。

每条记录代表一个业务模型（即一张业务表），包含：

- 模型基本信息（名称、集合名、页面编码等）
- 字段列表（每个字段的类型、显示规则、校验规则等）
- 布局配置（表单分组、字段排序等）
- 系统字段设置（默认排序字段、是否显示创建时间等）

当你在「模型管理」中新建或编辑一个模型并发布后，数据就写入这里。运行时页面（业务 CRUD 页）读取这里的配置来动态生成界面。

> 业务模型自身的权限不写在这里，而是写在 [`modmin_role_permissions`](#modmin_role_permissions)，按 `(roleCode, collectionName)` 为粒度。

---

### `modmin_admin_users`

存储管理后台账号。字段包括 `userName`、`nickName`、`passwordSalt`、`passwordHash`（SHA-256）、`roleCode`、`status`、`avatar` 等。新建用户的默认 `roleCode` 是 `role_operator`。

> 默认账号不在这里。早期 `modmin_auth` 内置过硬编码登录用于初始接入；正式环境应通过用户管理页面创建真实账号，不应依赖任何默认后台凭据。

---

### `modmin_admin_roles`

存储管理后台角色。系统内置两个不可删除的角色（由部署初始化流程显式补齐）：

| roleCode | 名称 | 说明 |
| --- | --- | --- |
| `role_super_admin` | 超级管理员 | 在所有云函数中硬编码旁路一切权限校验 |
| `role_operator` | 运营人员 | 默认无任何业务模型权限，需超管手动配置 |

可在「角色管理」页扩展自定义角色。`status` 为 `disabled` 时，5 个云函数都会拒绝该角色对应账号的鉴权与登录。

---

### `modmin_role_permissions`

存储角色与业务模型的增删改查权限矩阵，**是权限系统的唯一真源**。每条记录形如：

```json
{
  "roleCode": "role_editor",
  "collectionName": "articles",
  "canList": true,
  "canCreate": true,
  "canUpdate": false,
  "canDelete": false
}
```

`getMyPermissions` / `checkModelPermission` / `listCollectionSchemas` 等所有鉴权入口都查这张表。`role_super_admin` 不需要在这里写记录，会被代码旁路为全权限。

---

### `modmin_sessions`

存储登录 refresh token 会话。字段：`refreshToken`、`userId`、`expireTime`、`createTime`。

每次成功登录写入一条；`refreshToken` 接口会先删旧记录再写新记录（轮换）；`logout` 删除当前会话。access token 是 JWT 直接由 `MODMIN_JWT_SECRET` 验签，**不查这张表**，因此密钥轮换会让所有已签发 token 立即失效。

---

### `modmin_menus`

存储业务功能目录，对应前端左侧菜单的分组（如「内容管理」「用户管理」）。

系统首次初始化时会自动写入三个默认目录：

| 名称 |
| --- |
| 内容管理 |
| 用户管理 |
| 运营管理 |

创建模型时可以选择将其归属到某个目录下，决定它出现在菜单的哪个位置。

---

### `modmin_menu_groups`

存储菜单分组配置，用于对侧边菜单做更细粒度的分组管理（可选）。

---

## 附录 B：云函数说明

5 个云函数的 `exports.main` 入口都接受 `{ action, data, context, meta }` 形式的事件，鉴权信息从 `event.context.accessToken`（JWT，由 `MODMIN_JWT_SECRET` 签发）解析得到。所有云函数都遵循"角色 disabled 立即拒绝"的规则。

### `modmin_auth`

**负责登录认证**。需要开启 HTTP 触发器（前端首次登录时还没有 CloudBase 登录态，必须走 HTTP 调用）。

| action | 作用 |
| --- | --- |
| `login` | 校验账号密码 + 角色启用状态，签发 access/refresh token + CloudBase 自定义登录 Ticket |
| `refreshToken` | 轮换：删旧 refresh token，签新 access + refresh；同步重新校验角色启用状态 |
| `validateSession` | 仅做 JWT 验签，不查 DB |
| `getCurrentUser` | 从 access token 提取用户信息 |
| `logout` | 删除当前 refresh token 会话 |

> 错误码：`40101` 账号 / 密码错或用户停用、`40102` 登录态过期、`40103` 角色已停用。

---

### `modmin_schema`

**负责模型配置的读写**（模型管理工作台使用）。普通角色按 `modmin_role_permissions.canList` 过滤可见模型；超管看全部。

| action | 作用 | 鉴权 |
| --- | --- | --- |
| `listCollectionSchemas` | 模型列表 | 按 canList 过滤 |
| `getCollectionSchemaDetail` | 单个模型的完整配置（字段、布局） | 需 canList |
| `saveCollectionSchema` | 保存 / 发布模型 | 仅超管 |
| `deleteCollectionSchema` | 删除模型 | 仅超管 |
| `sortCollectionSchemas` | 调整模型排序 | 仅超管 |
| `assignMenuGroup` | 把模型批量归到指定菜单分组 | 仅超管 |
| `listBusinessDirectories` | 业务目录列表（与 modmin_system 同名 action 共存） | 全员可读 |

---

### `modmin_runtime`

**负责生成运行时页面配置**（业务 CRUD 页面使用）。

| action | 作用 | 鉴权 |
| --- | --- | --- |
| `getPageRuntimeSchema` | 根据 `pageCode` 读模型配置，聚合生成前端可直接消费的运行时结构（字段、权限、排序、布局） | 需 canList |

这是生成式 CRUD 页的核心接口，前端不直接操作原始模型配置，而是消费这里输出的派生结构。

---

### `modmin_crud`

**负责业务数据的增删改查**（所有通过模型管理创建的业务集合都走这里）。鉴权按 `(roleCode, collectionName, action)` 三元组在 `modmin_role_permissions` 中查记录。

| action | 作用 | 所需权限 |
| --- | --- | --- |
| `list` | 分页查询列表（含过滤、排序、软删除过滤） | canList |
| `detail` | 查询单条 | canList |
| `create` | 新建（自动注入系统保留字段） | canCreate |
| `update` | 更新（自动刷新 `modmin_updateTime` / `modmin_updateBy`） | canUpdate |
| `delete` | 软删除（标记 `modmin_isDeleted = true`，不物理删除） | canDelete |

所有业务记录会自动注入以下系统保留字段：

| 字段 | 含义 |
| --- | --- |
| `modmin_createTime` | 创建时间（毫秒时间戳） |
| `modmin_createBy` | 创建人 ID |
| `modmin_updateTime` | 最后更新时间 |
| `modmin_updateBy` | 最后更新人 ID |
| `modmin_isDeleted` | 是否已软删除 |
| `modmin_deleteTime` | 软删除时间 |
| `modmin_deleteBy` | 执行删除的人 ID |

---

### `modmin_system`

**负责系统配置管理**（角色、菜单、菜单分组、管理员账号、权限配置）。所有写操作都强制超管（`role_super_admin`）。

| action | 作用 |
| --- | --- |
| `listRoles` | 角色列表 |
| `saveRole` | 编辑 / 新建角色（拒绝以内置 roleCode 新建；内置角色 status 强制 enabled） |
| `getMyPermissions` | 当前用户视角的 `permMap`（前端鉴权数据源；超管返回全部模型；角色停用返回空 + `roleDisabled: true`） |
| `getRolePermissions` | 取指定角色的权限矩阵（用于权限配置抽屉） |
| `saveRolePermissions` | 批量 upsert 角色的权限矩阵 |
| `listAdminUsers` | 管理员账号列表 |
| `saveAdminUser` | 新建 / 编辑管理员（密码会重新计算 `passwordHash` + `passwordSalt`） |
| `disableAdminUser` | 停用管理员（status → `disabled`） |
| `deleteAdminUser` | 删除管理员 |
| `listBusinessDirectories` / `saveBusinessDirectory` | 业务目录管理（侧边一级菜单） |
| `listMenuGroups` / `saveMenuGroup` / `deleteMenuGroup` | 菜单分组管理（二级分组） |

---

## 附录 C：本地开发

如果你需要在本地调试云函数逻辑（修改后立即生效，无需上传），可以启动本地服务器代替云端执行。

### C.1 创建本地凭据文件

本地服务器需要腾讯云 API 密钥才能访问 CloudBase 数据库，并需要一个本地 JWT 密钥让云函数顶层校验通过。

1. 前往 [腾讯云控制台 → 访问管理 → API 密钥](https://console.cloud.tencent.com/cam/capi) 获取 SecretId 和 SecretKey
2. 用 `openssl rand -hex 32` 生成一个**本地专用的** JWT 密钥（务必和生产 `MODMIN_JWT_SECRET` 不一样）
3. 复制 `local-server/cloudbase.local.example.json`，创建 `local-server/cloudbase.local.json`：

```json
{
  "envId": "your-env-id",
  "secretId": "your-secret-id",
  "secretKey": "your-secret-key",
  "jwtSecret": "本地专用的 64 字符随机串"
}
```

> **安全提示**：`cloudbase.local.json` 已在 `.gitignore` 中忽略，**不要提交到代码仓库**。`local-server/index.js` 启动时会校验 `jwtSecret` 长度 ≥ 32，缺失或过短直接 `process.exit(1)`。

### C.2 启动本地服务

```bash
# 终端 1：启动本地云函数服务
cd local-server
npm install
npm run dev        # 使用 nodemon，改代码自动重启

# 终端 2：启动前端
# web-admin/.env.development 默认已配置 VITE_API_MODE=http
cd web-admin
npm run dev
```

本地服务器监听 `http://localhost:3100`，直接在本机执行云函数代码，数据库仍连接真实 CloudBase 环境。
